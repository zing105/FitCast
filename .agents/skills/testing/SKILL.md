# 테스트 필수 작성 스킬

이 스킬은 FitCast 프로젝트에서 기능 개발, 수정, 구현 작업 시 **반드시** 테스트를 작성하도록 규정합니다.

## 🔴 핵심 규칙

> **모든 코드 변경에는 반드시 해당하는 테스트가 동반되어야 합니다.**
> 테스트 없는 코드 변경은 완료된 것으로 간주하지 않습니다.

---

## 1. 단위 테스트 (Unit Test)

### 언제 작성하는가
- **새로운 유틸 함수** 생성 시
- **커스텀 훅** (hooks/) 생성 또는 수정 시
- **Zustand 스토어** (store/) 로직 변경 시
- **서비스 함수** (services/) 변경 시
- **데이터 변환/가공 로직** 변경 시

### 작성 위치
```
components/__tests__/    ← 컴포넌트 테스트
hooks/__tests__/         ← 훅 테스트
store/__tests__/         ← 스토어 테스트
utils/__tests__/         ← 유틸 테스트
```

### 작성 방법
- **도구**: Jest (Expo 기본 내장)
- **파일명**: `[대상파일명].test.ts` 또는 `[대상파일명].test.tsx`
- **최소 커버리지**: 주요 함수당 최소 2개 테스트 케이스 (정상 + 에지 케이스)

### 예시 구조
```typescript
// hooks/__tests__/useRecommendation.test.ts
import { useRecommendation } from '../useRecommendation';

describe('useRecommendation', () => {
  it('날씨 데이터가 없을 때 기본값을 반환해야 한다', () => {
    // ...
  });

  it('기온이 28도 이상이면 반팔/반바지를 추천해야 한다', () => {
    // ...
  });
});
```

### 실행 명령어
```bash
# 전체 단위 테스트
npx jest

# 특정 파일 테스트
npx jest --testPathPattern="useRecommendation"

# 워치 모드
npx jest --watch
```

## 2. E2E 테스트 (End-to-End Test)

### 언제 작성하는가
- **새로운 화면/페이지** 추가 시
- **사용자 플로우가 변경**되는 수정 시 (네비게이션, 폼 제출 등)
- **UI 레이아웃 변경** 시 (뷰포트별 깨짐 확인)
- **배포 전 최종 검증** 시

### 작성 방법
- **도구**: 브라우저 서브에이전트 (Playwright 기반)
- **대상**: 배포된 웹 앱 (`https://fitcast.pages.dev`)
- **뷰포트 목록** (최소 이 4개는 반드시 테스트):

| 뷰포트 | 기기 | 해상도 |
|--------|------|--------|
| iPhone SE | 작은 화면 | 375x667 |
| 카카오톡 인앱 | 축소 뷰포트 | 375x600 |
| iPhone 14 Pro | 일반 | 393x852 |
| Galaxy S21 | Android | 360x800 |

### E2E 테스트 체크리스트
각 화면 변경 시 다음을 반드시 확인합니다:

```
□ 상단 헤더/로고가 잘리지 않는가?
□ 하단 탭바(홈, 옷장, 마이) 글씨가 완전히 보이는가?
□ 스크롤 시 마지막 컨텐츠가 탭바에 가려지지 않는가?
□ 터치/클릭 영역이 겹치는 요소가 없는가?
□ 텍스트가 줄바꿈되어 UI를 깨뜨리지 않는가?
```

### E2E 실행 방법
브라우저 서브에이전트를 사용하여 다음과 같이 테스트합니다:

```
1. 배포된 URL에 접속
2. 각 뷰포트로 리사이즈
3. 스크린샷 캡처
4. 시각적 검증 수행
5. 결과 보고서 작성
```

## 3. 워크플로우 통합

### 기능 개발 시 흐름
```
1. 기능 구현 (코드 작성)
2. 단위 테스트 작성 및 실행 (npx jest)
3. 웹 빌드 및 배포 (npm run build:web)
4. E2E 브라우저 테스트 (4개 뷰포트)
5. 테스트 통과 확인 후 완료 처리
```

### 버그 수정 시 흐름
```
1. 버그 재현 (스크린샷 캡처)
2. 수정 코드 작성
3. 해당 버그를 검증하는 단위 테스트 추가
4. 빌드 및 배포
5. E2E 테스트로 수정 확인 (Before/After 스크린샷 비교)
```

---

## 4. 테스트 추가 시 주의사항

- **비동기 로직**: `async/await`과 `act()`를 사용하여 비동기 상태 변화를 올바르게 테스트
- **모킹**: 외부 API(Supabase, OpenWeatherMap, Gemini)는 반드시 모킹
- **스냅샷 테스트**: UI 컴포넌트는 스냅샷 테스트 고려하되 남용 금지
- **환경 분리**: `Platform.OS` 분기가 있는 코드는 웹/네이티브 모두 테스트

---

## 5. 테스트 미작성 시 알림

만약 시간이나 복잡성 등의 이유로 테스트를 작성하지 못한 경우:
1. `task.md`에 **미작성 사유**를 기록
2. 해당 항목에 `[ ] 테스트 작성 필요` 체크리스트 추가
3. 사용자에게 **notify_user**로 테스트 누락 사실을 명시적으로 알림
